Para o provisionamento da infraestrutura do Crafty Controller, utilizei a nuvem da Amazon Web Services (AWS), onde criei uma instância EC2 (Elastic Compute Cloud) dedicada, nomeada como crafty. O sistema operacional selecionado foi o Ubuntu Server 26.04 LTS (64-bit). Como especificação de hardware para suportar a aplicação e os servidores de jogos com estabilidade, adotei uma instância do tipo m7i-flex.large, equipada com 2 vCPUs e 8 GB de memória RAM. Para assegurar o acesso remoto à máquina, a autenticação foi estruturada por meio de um par de chaves criptográficas utilizando o algoritmo de alta segurança ED25519, com a chave privada exportada e salva no formato .ppk para conexões via SSH.

Na camada de segurança periférica da nuvem, configurei um Security Group para gerenciar de forma estrita o controle de tráfego de entrada (Inbound Rules). Foram liberadas as portas padrão para comunicações via SSH (22), HTTP (80) e HTTPS (443). Adicionalmente, foi necessário implementar uma regra de TCP Personalizado na porta 8443, que corresponde à porta nativa utilizada pelo painel web do Crafty Controller. Essa regra foi definida com o range de origem global 0.0.0.0/0, o que viabiliza o acesso externo à interface gráfica de gerenciamento a partir de qualquer endereço IP na internet, eliminando restrições geográficas de acesso para os administradores.

Após a consolidação do ambiente de rede, realizei a instalação do ecossistema seguindo os scripts e diretrizes contidos na documentação oficial do Crafty Controller. Visando adotar as melhores práticas de administração de sistemas (SysAdmin), apliquei um refinamento na pós-instalação: a aplicação foi isolada dentro de um ambiente virtual de Python (.venv) e configurada como um daemon nativo do Linux via systemd. Essa abordagem garantiu que o painel passasse a rodar de forma resiliente em segundo plano (background) sob um usuário de menores privilégios (crafty), contando com políticas de autorrecuperação (on-failure) e inicialização automatizada diretamente no boot da instância AWS, eliminando a dependência de sessões ativas de terminal.

Com o intuito de otimizar a experiência do usuário e implementar conceitos avançados de roteamento de tráfego, configurei um servidor web Nginx atuando como Proxy Reverso. Essa arquitetura permitiu o desacoplamento de portas, mapeando o subdomínio crafty.lucascruz.shop para interceptar requisições nas portas web padrões (80/443) e encaminhá-las internamente para a porta de gerenciamento (8443), com terminação SSL via Certbot (Let's Encrypt). Em paralelo, aproveitando a capacidade do Nginx de segregar tráfego por protocolos e hosts, o subdomínio principal de conexão do jogo (mc.lucascruz.shop) foi configurado para responder de forma dupla: enquanto atua como ponte direta para o protocolo TCP do servidor de Minecraft na porta nativa 25565, ele serve uma aplicação client-side customizada em HTML/JavaScript nas portas HTTP para requisições vindas de navegadores web, validando o isolamento completo de camadas da aplicação.

Para complementar o monitoramento da infraestrutura, implementei uma API REST de telemetria baseada no framework Spring Boot e Java 21, com o objetivo de centralizar e expor os dados vitais do servidor de jogos de forma performática. A arquitetura da aplicação foi desenhada utilizando o padrão DTO (Data Transfer Object) para desacoplamento de dados e serialização automática de objetos Java em formato JSON. O consumo de dados brutos foi estruturado por meio do componente RestTemplate, realizando requisições a serviços externos de checagem de estado para alimentar o modelo interno da aplicação de forma dinâmica e defensiva contra falhas de conexão.

Na camada de distribuição e rede, integrei a API ao ecossistema do Proxy Reverso do Nginx dentro do bloco de servidor seguro (HTTPS) do subdomínio principal (mc.lucascruz.shop). Essa configuração mapeia cirurgicamente o endpoint `/api/status` e encaminha o tráfego interno para a porta nativa 8080 gerenciada pelo servidor Tomcat embutido. Visando garantir a resiliência e a alta disponibilidade do microsserviço, a aplicação foi encapsulada como um daemon nativo do Linux via systemd, contando com políticas de autorrecuperação automática (always restart) em cenários de crash e persistência ativa em segundo plano, tornando o microsserviço totalmente independente de sessões ativas de terminal SSH.
